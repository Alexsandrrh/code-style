# Code Style

Этот документ описывает базовые договорённости по оформлению React-кода, чтобы новые участники команды могли быстро включиться в работу.

## Именование файлов и папок

Сохраняйте единый стиль, чтобы структура проекта оставалась предсказуемой для всей команды.

**Правила по умолчанию**
- Именуйте директории и файлы в `kebab-case`: `my-component`, `my-component/button.tsx` — так пути легко читаются на любой платформе.
- Создавайте `index.ts` на уровне директории, если нужно собрать экспорт в одной точке.
- Выносите дочерние компоненты и хуки в подпапки `components/` и `hooks/`, чтобы разгрузить корневой каталог компонента.
- Не переименовывайте служебные файлы инструментов (`package.json`, `tsconfig.json` и т. д.), иначе сторонние скрипты их не найдут.

**Исключения**
- Файлы и папки, обслуживающие роутинг, можно именовать в `camelCase`, если этого требует маршрутизатор.
- `index.tsx` или `index.jsx` допустимы только там, где их использует роутинг; остальные компоненты называйте явно.

**Пример структуры**
```text
src/
├── components/
│   ├── index.ts
│   └── user-card/
│       ├── index.ts
│       ├── user-card.tsx
│       ├── user-card.module.css
│       ├── user-card.test.tsx
│       ├── hooks/
│       │   ├── index.ts
│       │   └── use-user-card.ts
│       └── components/
│           ├── index.ts
│           └── user-card-avatar.tsx
└── pages/
    └── dashboard/
        ├── index.tsx
        └── dashboard.tsx
```

## Комментарии в коде

Комментарии дополняют код, а не дублируют его, поэтому держите их краткими и актуальными.

**Когда пишем**
- Поясняйте только неочевидные решения: фиксируйте намерения, ограничения и исторический контекст.
- Регулярно пересматривайте комментарии и удаляйте устаревшие заметки.

**Формат JSDoc для публичного API**
- Используйте блок `/** ... */` с лаконичным `@description`; добавляйте `@deprecated`, если элемент нужно избегать.
- Завершайте каждое предложение точкой, чтобы текст выглядел аккуратно.
- Размещайте JSDoc непосредственно над полем интерфейса или сущностью, чтобы IDE показывала подсказку.
- Добавляйте теги `@param`, `@returns` и другие только при реальной необходимости.

**Локальные пояснения**
- Короткие комментарии внутри функций пишите как `//` на отдельной строке.

**Пример интерфейса**
```typescript
export interface ComponentProps {
  /**
   * @description Тестовый пропс для примера.
   */
  testProps: unknown;
}
```

**Пример использования внутри компонента**
```typescript
/**
 * @description Компонент, отображающий карточку пользователя.
 */
export const Component: FC<ComponentProps> = () => {
  // Заглушка до появления реальной разметки.
  return null;
};
```

## React-компоненты

Держите API компонентов прозрачным и документированным, чтобы их было легко использовать и поддерживать.

**Контракт компонента**
- Описывайте публичные свойства через `interface ComponentProps`, чтобы типы оставались явными.
- Если компонент полностью повторяет чужой контракт и не добавляет полей, используйте алиас `type ComponentProps = AnotherProps;`, чтобы не плодить пустые интерфейсы.

**Объявление и структура**
- Объявляйте компонент как `const Component: FC<ComponentProps>`, чтобы IDE подсказывала пропсы и `children`.
- Используйте `null` как временную заглушку до появления разметки.
- Присваивайте `Component.displayName`, чтобы анонимные функции не мешали отладке в React DevTools.

**Документация**
- Сопровождайте интерфейс и компонент JSDoc-описаниями в соответствии с правилами раздела «Комментарии».

**Минимальный шаблон**
```typescript
import type { FC } from "react";

/**
 * @description Свойства компонента.
 */
export interface ComponentProps {}

/**
 * @description Компонент-заглушка.
 */
export const Component: FC<ComponentProps> = () => null;

Component.displayName = "Component";
```
